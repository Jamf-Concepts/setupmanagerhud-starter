## Project Overview

Setup Manager HUD is a real-time webhook dashboard for [Setup Manager](https://github.com/jamf/setup-manager). This is a **starter template** that wraps the `@motionbug/setupmanagerhud-core` npm package and deploys to Cloudflare Workers.

## Commands

```bash
npm run dev      # Start local development server (wrangler dev)
npm run sync     # Clean and sync assets/migrations from the core package
npm run deploy   # Apply D1 migrations and deploy to Cloudflare Workers
npm run upgrade  # Update core package and sync new assets
```

Set the webhook token secret:
```bash
npx wrangler secret put WEBHOOK_TOKEN
```

## Critical Deploy Button Requirement

`WEBHOOK_TOKEN` is required for every usable deployment. The Cloudflare Deploy
Button must show an interface that asks the user for this secret during setup.

Do not remove `.dev.vars.example`: Cloudflare uses this file to discover Worker
secrets for the Deploy Button flow. Keep `WEBHOOK_TOKEN=` in that file so users
are prompted to paste their own generated token rather than inheriting a shared
placeholder.

Do not remove the `[secrets] required = ["WEBHOOK_TOKEN"]` declaration from
`wrangler.toml`. It makes manual deploys fail clearly when the token has not
been configured.

The same generated token must be saved by the user and configured in Setup
Manager. Cloudflare Access is optional dashboard protection; it does not replace
the required webhook token.

## Architecture

This is a thin deployment wrapper around `@motionbug/setupmanagerhud-core`:

- **src/index.ts** - Entry point that re-exports the core app and DashboardRoom Durable Object
- **wrangler.toml** - Cloudflare Workers configuration with D1 database and Durable Objects bindings
- **public/** and **migrations/** - Synced from the npm package (gitignored, regenerated by `npm run sync`)

The starter currently depends on `@motionbug/setupmanagerhud-core@^1.2.5`.
`npm run sync` intentionally removes and regenerates `public/` and
`migrations/` so stale hashed dashboard JavaScript cannot remain after an
upgrade.

Core `1.2.5` includes the Active/Disabled record visibility fix. Active is the
normal dashboard view. Disabled records are hidden from the Active record list
but remain stored in D1 and can be enabled again. Do not treat Disabled as
deletion.

Do not reintroduce dashboard mutation routes shaped like
`/api/events/:id/archive` or `/api/events/:id/disabled`. Those URLs were
blocked by browser content blockers. Dashboard visibility behavior should use
the neutral records API provided by the core package, such as
`/api/records/visibility`.

### Cloudflare Bindings

- `DB` - D1 database for event persistence
- `DASHBOARD_ROOM` - Durable Object for real-time WebSocket connections
- `WEBHOOK_TOKEN` - Required shared secret for webhook authentication; must be exposed in the Deploy Button setup via `.dev.vars.example`
- Optional: `CF_ACCESS_AUD` and `CF_ACCESS_TEAM_DOMAIN` for Cloudflare Access protection

### Customization

The entry point can be extended by wrapping `app.fetch` or adding custom routes before delegating to the core handler.
